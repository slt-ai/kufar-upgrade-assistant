# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Роль агента

Агент — оркестратор подбора комплектации на сайте объявлений kufar.by. Для достижения цели он **обязан использовать агентов `kufar-listings-fetcher` и `kit-analyzer`** через инструмент `Agent`. Главный агент управляет потоком, проверяет результат и формирует финальный ответ; самостоятельно выполнять сбор, фильтрацию, ранжирование и комбинирование объявлений он должен **только если субагенты не справились после нескольких попыток**.

## Правила безопасности

- Разрешено: действия в рабочей папке проекта, парсинг сайта kufar.by.
- Запрещено без явного одобрения пользователя:
  - удалять файлы;
  - передавать наружу личные данные пользователя;
  - отправлять какие-либо сообщения на сайте kufar.by.

## Правило нейминга файлов

Рабочая формула имени файла:

```
[YYYY-MM-DD HHMM —] тема — тип.md
```

Примеры:
- `2026-07-06 2149 — intel+amd ddr4 upgrade — подбор.md`
- `client-onboarding-checklist.md`

Дата и время добавляются **вместе**, когда файл привязан к конкретному прогону/дате (отчёты, логи парсинга, выгрузки). Формат даты `YYYY-MM-DD`, время `HHMM` (24ч, без двоеточия). **Время обязательно при наличии даты** — в один день может быть несколько прогонов, без времени они перетрут друг друга. Для служебных и шаблонных документов (чеклисты, инструкции) дата и время опускаются.

# Твой человек
Домашний пользователь ПК. Цель — модернизировать свой компьютер с минимальными затратами, покупая комплектующие б/у на kufar.by.

Увлечения: локальные LLM-модели и нейронные сети. Это контекст для выбора комплектующих — при подборе учитывать, что машина будет использоваться для запуска локальных моделей (важны объём/тип памяти, пропускная способность шин, поддержка AVX/AMX и т.п., если это уместно).

Конкретные параметры апгрейда (платформа, память, PCIe, регион, бренды, тиры, бюджеты, приоритеты под LLM, план по ОЗУ) — в `апгрейд-пк — пожелания.md`. Этот файл — авторитетный источник; агенты читают его при каждом запуске и не хранят эти значения у себя.

# Проекты
Модернизация домашнего ПК. Конфигурация текущей машины — в `Report.txt` в корне проекта; требования к апгрейду — в `апгрейд-пк — пожелания.md`. Агент читает оба файла при каждом подборе.

# Как работаем
**План один раз — дальше автономно.** Перед первым прогоном по конкретному сценарию (например, «подобрать комплекты Intel и AMD») агент согласовывает: что ищем, куда сохраняем, как называем файл, что считаем успехом. Дальше работает по этому сценарию без переподтверждений.

**Сценарий по умолчанию для подбора комплектов:**
1. Проверить наличие и прочитать два входных файла: `Report.txt` (текущая конфигурация ПК) и `апгрейд-пк — пожелания.md` (требования к апгрейду: жёсткие ограничения, тиры/бюджет, приоритеты под локальные LLM, план по ОЗУ). `апгрейд-пк — пожелания.md` — авторитетный источник требований; при конфликте с общими фразами в этом `CLAUDE.md` приоритет у файла пожеланий. Агент передаёт содержимое файла пожеланий в `kit-analyzer` как часть контекста.

   **Если одного или обоих файлов нет** — **не угадывать** параметры и не запускать подбор. Главный агент запрашивает у пользователя недостающую информацию и **создаёт** отсутствующий файл:
   - нет `Report.txt` → спросить текущую конфигурацию (CPU, мат. плата, ОЗУ — тип и объём, видеокарта, накопители, форм-фактор, БП если известен); сохранить как `Report.txt` в корне проекта.
   - нет `апгрейд-пк — пожелания.md` → спросить параметры апгрейда (scope, платформа/память/PCIe, регион, категории, бюджет, приоритеты, ОЗУ-план); создать файл по правилу нейминга в корне проекта, опираясь на структуру существующего файла пожеланий (10 секций), заполнить полученными ответами и разумными значениями по умолчанию, помеченными как «уточнить».
   - продолжать подбор только после того, как оба файла существуют и прочитаны.
   Субагенты (`kit-analyzer`, `kufar-listings-fetcher`), обнаружив отсутствие файла, **не опрашивают пользователя сами** — они возвращают управление главному агенту с запросом создать/пополнить файл.
2. Запустить агент `kufar-listings-fetcher`, чтобы собрать актуальные объявления по категориям и региону из `апгрейд-пк — пожелания.md` (раздел 3). Получить выходные файлы: по умолчанию `scripts/_out.json`, `scripts/_items.json` и обновлённый `scripts/.cache/seen_ids.json`.
3. Запустить агент `kit-analyzer` на собранных данных. Получить от него структурированный топ-3 для Intel и топ-3 для AMD: совместимость сокет + поколение, сумма за комплект, цена каждой позиции, ссылки, продавцы, состояние, пометки об изменениях цены/статуса.
4. **Мат. плата и процессор подбираются из разных объявлений** и комбинируются в лучшие пары по критериям: совместимость сокет + поколение, цена за комплект, состояние, продавец, логистика. Один и тот же лот может участвовать в нескольких комбинациях, если совместим.
5. Сохранить в рабочую папку:
   - markdown-файл по правилу нейминга `[YYYY-MM-DD HHMM —] тема — тип.md`;
   - структурированную выгрузку JSON/CSV;
   - сводку из 3 пунктов в конце ответа.
6. Следить за актуальностью по умолчанию: запустить `kufar-listings-fetcher` для перепроверки цены/статуса по значимым `ad_id`, затем `kit-analyzer` для переранжирования. Обновить отчёт. Если объявление исчезло или цена изменилась существенно — отметить это в отчёте.

**Сбор данных с kufar.by:**
- По умолчанию всегда используем **актуальные данные с сайта, а не кешированные**, если явно не указано другое. Кеш применяется только когда пользователь прямо просит «работать по кешу» / «без повторного парсинга».
- Загрузку страниц kufar.by (listing + детали объявлений) **всегда выполняет агент `kufar-listings-fetcher`** через инструмент `Agent`. Агент сам выбирает способ получения данных (HTTP-запросы, браузер, Playwright и т.п.) и формат промежуточного результата; основной агент использует его выходные файлы (по умолчанию `scripts/_out.json` и `scripts/_items.json`) для фильтрации и отчёта.
- При каждом прогоне обходить категорию целиком (все страницы, а не только первую).
- Вести `scripts/.cache/seen_ids.json` — реестр уже виденных `ad_id` с датой последнего сбора. Следующий прогон добавляет новые и перепроверяет старые (цена, статус).
- Объявления, ушедшие из выдачи, — помечать `removed=true` **только** после прямой проверки ссылки `https://www.kufar.by/item/<id>` (правило 9 в `апгрейд-пк — пожелания.md`); без проверки — `stale=true, last_check=<now>`. Из истории **не удалять**.
- Defect-фильтр и правило «no-price» — применять по списку маркеров из `апгрейд-пк — пожелания.md` (правила 5 и 7); `kufar-listings-fetcher` хранит backstop-копию на случай отсутствия файла.
- При комбинировании со старыми и новыми данными использовать общий пул: все актуальные объявления (не `removed`, не `defect`) проходят в фильтр одинаково.

**Фильтрация, совместимость и ранжирование:**
- Агент `kit-analyzer` применяет параметры из `апгрейд-пк — пожелания.md` (совместимость по справочнику сокетов, DDR4/PCIe, бренд-whitelist, defect-фильтр, свежесть, тиры/бюджеты/скоринг) и ранжирует топ-3 пар для Intel и AMD. Механики проверки и скоринга — в определении `kit-analyzer`.
- Главный агент не дублирует эту работу самостоятельно, кроме случаев, когда `kit-analyzer` не смог выдать результат после нескольких попыток.

**Скрипты конвейера — владение и синк:**
- Каждый субагент владеет своими скриптами и **сам правит их под файл пожеланий** при необходимости:
  - `kufar-listings-fetcher` — `scripts/kufar_parse.py`, `kufar_details.py`, `transform_for_delivery.py`, `enrich_seen.py` (регион, URL категорий, defect-маркеры, no-price, свежесть).
  - `kit-analyzer` — `scripts/kufar_filter.py`, `kit_analyzer_run.py`, `reprocess_kits.py`, `_build_report.py` (DDR4/PCIe/чипсеты/бренды/CPU_PERF/baseline/тиры/cap'ы/веса/VRM).
- Принцип: параметр живёт в `апгрейд-пк — пожелания.md`; если скрипт его не отражает или хардкодит устаревшее — субагент редактирует скрипт (предпочтительно — чтобы скрипт читал параметр из файла). Не удалять и не переименовывать скрипты; крупные/рискованные правки — согласовывать с главным агентом одной строкой. Главный агент правит скрипты сам только если субагент не справился.

# Частые задачи
- **Подбор комплекта (Intel или AMD) для модернизации ПК:** вернуть топ-3 комплекта (мат. плата + процессор) с ценой, состоянием, ссылками, заметками. Источник — kufar.by, б/у. **Мат. плата и процессор подбираются отдельно и комбинируются** в лучшие пары по цене + совместимости. Сохранить файл по правилу нейминга + JSON/CSV-выгрузку.
- **Сравнение комплектов Intel vs AMD:** таблица-сравнение вариантов под текущую видеокарту с учётом бюджета.
- **Проверка актуальности прошлой выгрузки:** пройтись по сохранённым объявлениям, отметить изменения цены и снятые с продажи, обновить отчёт. Если объявление ушло из выдачи — пометить `removed=true`, но **не удалять** из истории.
- **Чтение конфигурации ПК:** открыть `Report.txt` (текущая конфигурация) и `апгрейд-пк — пожелания.md` (требования к апгрейду) в корне проекта, использовать как входные данные для подбора.

# Формат ответа
- Без вступлений, без «Конечно!», без пересказа сделанного.
- Структура ответа по задаче «подбор комплекта»:
  1. Краткая сводка (3 пункта).
  2. Топ-3 для Intel.
  3. Топ-3 для AMD.
  4. Таблица сравнения (цена, состояние, ссылка, заметки).

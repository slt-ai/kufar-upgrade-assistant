# /kufar-upgrade — подбор комплекта на kufar.by Минск

Запускает конвейер подбора мат. платы + процессора под указанную платформу (Intel LGA1700/DDR4 или AMD AM4/DDR4) для домашнего ПК с видеокартой RTX 5060 Ti. Кросс-комбинирование: мат. плата и процессор подбираются из разных объявлений.

## Категории kufar с преднастроенными фильтрами

Для поиска **материнских плат** использовать только эти ссылки (они передают фильтры через `ccch`/`ccrt`/`sort`):

- **Intel LGA1700/1200 (DDR4):**
  `https://www.kufar.by/l/r~minsk/materinskie-platy?ccch=v.or%3A32%2C31%2C75%2C55%2C74%2C54%2C53&ccrt=v.or%3A4&sort=lst.d`
- **AMD AM4 (B550/X570, DDR4):**
  `https://www.kufar.by/l/r~minsk/materinskie-platy?ccch=v.or%3A21%2C11&ccrt=v.or%3A4&sort=lst.d`

Для поиска **процессоров** использовать стандартную категорию:
- `https://www.kufar.by/l/r~minsk/processory`

> **Важно:** `scripts/kufar_parse.py` принимает URL аргументами и **обязан сохранять query** (см. memory `project-kufar-parse-url-fix`). Если меняется `kufar_parse.py` — проверять, что `first_url` строится через `urllib.parse.urlunparse(..., parsed.query, ...)`, а не обрезается до `scheme://netloc/path`.

> **Режим сбора:** `kufar_parse.py` по умолчанию работает **инкрементально** — читает `scripts/.cache/_last_run.json` и собирает только объявления с `list_time` позже последнего прогона (минус 5 мин запас), останавливая пагинацию, когда верхнее объявление на странице старее cutoff. Свежие объявления мержатся в существующий `scripts/_out.json` (новые перезаписывают старые по id), а дельта пишется в `scripts/_out_delta.json`. Старые лоты при этом не перепроверяются — снятие/смену цены ловит только отдельная priority-проверка. Передай `--full`, чтобы принудительно сделать полный обход и пересобрать `_out.json` с нуля (нужно при первом запуске, при смене фильтров/сортировки, или если подозреваешь, что старые лоты устарели). Инкрементальный режим опирается на сортировку kufar «свежие сверху» — не используй его с URL, где задан не-временной `sort=`.

## Когда применять

- Пользователь просит «подобрать плату + процессор», «intel+amd», «обнови отчёт», «собери комплект» и т.п.
- Не применять для AMD AM5 (DDR5), Intel LGA1851 — это вне текущего сценария (CLAUDE.md).

## Что делает

1. Прочитать `Report.txt` в корне проекта — определить текущую конфигурацию (объём RAM, чипсет, GPU).
2. Проверить, когда был последний успешный прогон:
   - `ls -la scripts/_items.json` → mtime.
   - Если < 30 минут: **не парсить kufar заново** — работать по `_items.json` (так и просил пользователь 2026-06-10: «не дёргать kufar заново»).
   - Если > 30 минут: спросить пользователя «парсить заново или работать по кешу?». По умолчанию — кеш.
3. Прогнать конвейер: `kufar_filter.py` → `_build_report.py`.
4. Если есть новые данные и пользователь дал добро на полный прогон — `kufar_parse.py` → `kufar_details.py` → `kufar_filter.py` → `_build_report.py`. **URL для парсинга — из секции «Категории kufar с преднастроенными фильтрами» выше.** Парсить **обе** ссылки на мат. платы, передавая их аргументами в `kufar_parse.py`.
5. Сохранить результат в `reports/` по правилу нейминга: `YYYY-MM-DD HHMM — тема — подбор.md` (тема: `intel+amd ddr4 upgrade`, `intel ddr4 upgrade`, `amd ddr4 upgrade`, `mb-only intel+amd ddr4` и т.п.). **Время HHMM обязательно** — в один день может быть несколько прогонов, без времени они бы перетирали друг друга. Пример: `2026-06-12 0043 — mb-only intel+amd ddr4 — подбор.md`.
6. Создать зеркальный JSON: `reports/YYYY-MM-DD HHMM — тема — подбор.json` (то же имя, другое расширение).
7. Исключить noname-бренды (см. memory `project-mb-brand-policy`): whitelist = {ASUS, MSI, Gigabyte, ASRock, Biostar, ECS}. Остальные бренды помечать `noname=true` в выгрузке и **не включать в топ-3**.
8. Соблюдать стандартные фильтры (см. CLAUDE.md):
   - DDR4 only;
   - Intel: LGA1700 + чипсеты Z690/Z790 (PCI-E 5.0) или B660/B760/H610/H670 (PCI-E 4.0);
   - AMD: AM4 + чипсеты B550/X570 (PCI-E 4.0);
   - регион: Минск;
   - дефекты (`defect_marker`) — исключать (в `kufar_details.py` маркеры: «не работает», «дефект», **«дифект»** (опечатка продавца), «слома», «убит», «вышел из строя», «трещина», «глючит» и т.п.);
   - **лоты без числовой цены («договорная», «торг», «negotiable», «обмен») — исключать** (см. memory `project-kufar-no-price-skip`); их нельзя ранжировать по `total`;
   - объявления с `removed=true` — не трогать, но не использовать в подборе.
9. В финальном ответе:
   - Сводка из 3 пунктов;
   - Топ-3 для запрошенной платформы (или для обеих);
   - Таблица сравнения (если обе платформы);
   - Список созданных/обновлённых файлов.

## Входные данные (аргумент)

`intel` | `amd` | `both` (по умолчанию `both`).

Парсится как первый позиционный аргумент. Если ничего не передано — `both`.

## Файлы

- Читает: `Report.txt`, `scripts/_items.json`, `scripts/_candidates.json`, `scripts/_out.json`, `.cache/seen_ids.json`.
- Пишет: `reports/YYYY-MM-DD HHMM — тема — подбор.md`, `reports/YYYY-MM-DD HHMM — тема — подбор.json` (время `HHMM` обязательно), при необходимости — обновляет `scripts/_items.json` и `.cache/seen_ids.json`.
- Не удаляет файлы без явного одобрения.

## Границы

- **Не удалять** файлы (правило проекта).
- **Не отправлять** ничего на kufar.by (правило проекта).
- **Не выходить** за пределы `D:\Work\AI\ClaudeCode\research-agent`.
- **Не переименовывать** существующие файлы.
- **Не запускать** `kufar_parse.py` / `kufar_details.py` без явного подтверждения от пользователя, если последний прогон был < 30 минут назад.

## Что уточнять у пользователя перед прогоном

1. Если последний успешный парс > 30 минут: «парсить заново или работать по кешу?». По умолчанию — кеш.
2. Платформа: `intel` / `amd` / `both` (если не указана в вызове).
3. Имя файла отчёта (если пользователь хочет нестандартное — иначе `YYYY-MM-DD HHMM — тема — подбор.md`).
4. Формат выгрузки: `json:топ-пары` (по умолчанию), `json:все-объявления`, `csv:комплекты`, `не выгружать`. По умолчанию — `json:топ-пары`.

## Сценарий по умолчанию (план)

1. Прочитать `Report.txt` (если не читан в этой сессии).
2. Проверить `scripts/_items.json` mtime.
3. Если < 30 мин: скип парса, сразу `kufar_filter.py` → `_build_report.py`.
4. Если > 30 мин: спросить пользователя.
5. Сформировать markdown-отчёт по шаблону (см. сегодняшний `reports/2026-06-10 — intel+amd ddr4 upgrade — подбор.md`).
6. Сформировать JSON-выгрузку.
7. Сводка из 3 пунктов.

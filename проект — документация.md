# Проект — документация

Справочник по устройству проекта «research-agent» — подборка комплектации для
апгрейда домашнего ПК на основе объявлений kufar.by. Для сопровождения, поддержки
и изменений. Дата состояния: 2026-07-07.

---

## 1. Назначение

Пользователь модернизирует домашний ПК с минимальными затратами, покупая б/у
комплектующие на kufar.by (Минск). Текущий сценарий — замена мат. платы + CPU под
видеокарту RTX 5060 Ti (нужна шина PCI-E 4.0/5.0 x16), staying на DDR4. Агент
собирает объявления, фильтрует по совместимости, ранжирует топ-3 комплекта для
Intel и AMD, сохраняет отчёт + JSON-выгрузку, публикует HTML на внутренний сайт.

Контекст использования машины — локальные LLM-модели (учитывается в приоритетах
подбора: генерация на GPU → важна шина под видеокарту, а не CPU-фичи).

---

## 2. Архитектура: SSOT из двух файлов + generic-агенты

С 2026-07-06 система перестроена под **единый источник параметров**:

- **`Report.txt`** — текущая конфигурация ПК (CPU, плата, ОЗУ, GPU, накопители, форм-фактор).
- **`апгрейд-пк — пожелания.md`** — все параметры апгрейда (10 секций: scope, анкор,
  жёсткие ограничения, регион+URL категорий, справочник сокет-совместимости, тиры+бюджет,
  параметры скоринга, приоритеты под LLM, план по ОЗУ, фокус).

Главный агент, `kit-analyzer` и `kufar-listings-fetcher` **читают эти файлы при каждом
запуске и не хранят значения параметров в своих определениях**. Правка любого из двух
файлов → система ищет по новым параметрам без правки агентов.

**Если одного или обоих файлов нет** — подбор не запускается, параметры не угадываются.
Главный агент опрашивает пользователя и создаёт недостающий файл. Субагенты при
отсутствии файла возвращают управление главному агенту (не опрашивают пользователя сами).

В `kufar-listings-fetcher` осталась **backstop**-копия defect-маркеров и no-price —
safety-дубль на случай отсутствия/порчи файла пожеланий. При наличии файла приоритет у
него; backstop — last-resort.

> ⚠️ **Важно (см. секцию 14):** SSOT-перестройка пока покрыла только agent-файлы
> (`CLAUDE.md`, `kit-analyzer.md`, `kufar-listings-fetcher.md`) и skills-описания. Python-
> скрипты конвейера всё ещё хардкодят часть параметров. Реальное поведение при правке
> файла пожеланий зависит от того, какой путь запущен — agent-driven или script-driven.

## 4. Входные файлы

### `Report.txt`
AIDA64-дамп текущей машины. Ключевое для подбора: CPU (baseline скоринга), тип/объём ОЗУ,
видеокарта (от неё считается требуемая PCIe), текущая плата (что апгрейджаем), форм-фактор.
Файл с фиксированным именем — не подчиняется формуле нейминга.

### `апгрейд-пк — пожелания.md`
10 секций (см. секцию 2 выше). Это **авторитетный источник** — при конфликте с мягкими
формулировками в `CLAUDE.md` приоритет у этого файла. Секция 4 («Справочник совместимости»)
— факты железа (сокет→поколение→чипсет→память→PCIe), агенты читают, пользователь редко правит.
Остальные секции — пользовательские пожелания, правятся под новый сценарий.

## 5. Агенты

### Главный агент (оркестратор) — `CLAUDE.md`
Управляет потоком: читает два входных файла, запускает субагентов, проверяет результат,
формирует финальный ответ пользователю. Сам выполняет сбор/фильтрацию/ранжирование **только**
если субагенты не справились после нескольких попыток. Владеет общением с пользователем
(в т.ч. создание недостающих входных файлов).

### `kufar-listings-fetcher` — `.claude/agents/kufar-listings-fetcher.md`
Сбор raw-данных с kufar.by. Параметры (регион, URL категорий, defect-маркеры, no-price,
свежесть, removed/stale) — из файла пожеланий; backstop defect/no-price — в определении.
Механики: полная пагинация, incremental-режим, rate-limit, кеш `seen_ids.json`, field-schema.
Не парсит kufar напрямую без запроса, если прогон свежий (< 30 мин — политика skill'а).

### `kit-analyzer` — `.claude/agents/kit-analyzer.md`
Второй проход по собранным данным → shortlist валидированных совместимых свежих пар.
Параметры (DDR4/PCIe/регион/бренды/defect/сокет-таблица/тиры/cap'ы/пороги/веса/VRM-штраф/
baseline) — из файла пожеланий. Механики (в определении): процедура проверки совместимости,
формула скоринга, дедуп между тирами, JSON-shape вывода, freshness-procedure. Read-only на
листинги — не парсит kufar. Возвращает JSON-shortlist главному агенту.

## 6. Skills

### `/kufar-upgrade` — `.claude/skills/kufar-upgrade/SKILL.md`
Полный цикл подбора: `kufar_parse.py → kufar_details.py → kufar_filter.py → _build_report.py`.
Аргумент: `intel` | `amd` | `both` (по умолчанию `both`). Проверяет mtime `_items.json`:
если < 30 мин — не парсит заново, работает по кешу. Сохраняет `reports/YYYY-MM-DD HHMM —
тема — подбор.{md,json}`. ⚠️ В SKILL.md всё ещё хардкодятся URL с преднастроенными фильтрами
(`ccch`/`ccrt`/`sort`) и часть параметров — см. секцию 14.

### `/kufar-refresh` — `.claude/skills/kufar-refresh/SKILL.md`
Точечная перепроверка сохранённых в прошлом отчёте объявлений (цена/статус/наличие) по
прямым ссылкам `https://www.kufar.by/item/<id>`. Не запускает полный парс. Помечает
`removed=true`/`price_changed`, обновляет `seen_ids.json` (не удаляет), перегенерирует отчёт.
Аргумент — имя отчёта (по умолчанию самый свежий `reports/*-подбор.md`).

## 7. Конвейер сбора и подбора (scripts/)

Энд-ту-энд (script-driven путь, через `/kufar-upgrade`):

```
kufar_parse.py   →  _out.json (+_out_delta.json, .cache/_last_run.json, seen_ids.json)
kufar_details.py →  _items.json (детали + defect-флаги; incremental по кешу .cache/items/)
kufar_filter.py  →  _candidates.json ({intel_mb, amd_mb, intel_cpus, amd_cpus})
_build_report.py →  reports/YYYY-MM-DD HHMM — тема — подбор.{md,json}
```

Дополнительно:
- `kit_analyzer_run.py` / `reprocess_kits.py` — альтернативные ранжировщики с CPU_PERF-
  таблицей (baseline i7-9750H = 100) и трёхтирным скорингом (entry/balanced/future_ready).
- `enrich_seen.py` — проставляет defect/no_price/stale в `seen_ids.json` по `_candidates.json`.
- `transform_for_delivery.py` — плоская схема доставки (ad_id, url, title, body, price,
  currency, condition, seller, location, listing_time, category, source_page, collected_at, images).
- `md_to_html.py` — `.md` → standalone `.html` (CSS inline, тёмная тема, скроллящиеся таблицы).
- `sync-to-caddy.sh` — публикация `*.html` на Caddy LXC.

Incremental-режим `kufar_parse.py`: по умолчанию собирает только лоты новее
`_last_run.json` (минус 5 мин запас), останавливает пагинацию, когда верхний лот на
странице старее cutoff, мержит в `_out.json`. `--full` — полный пересбор с нуля (первый
прогон, смена фильтров/сортировки, подозрение на устаревание). Опирается на сортировку
kufar «свежие сверху» — не применять к URL с не-временным `sort=`.

> ⚠️ `kufar_parse.py` обязан сохранять query в URL (фикс `project-kufar-parse-url-fix`):
> `first_url` строится через `urllib.parse.urlunparse(..., parsed.query, ...)`. При правке
> скрипта проверять, что query не обрезается — иначе фильтры `ccch`/`ccrt`/`sort` теряются.

## 8. Кеш и реестр `seen_ids.json`

`scripts/.cache/seen_ids.json` — `ad_id` → `{first_seen, last_seen, last_list_time,
last_price, last_check, removed, defect, no_price, stale}`.

Правила статусов (параметры — в файле пожеланий, секция 2):
- `removed=true` — **только** после прямой проверки `https://www.kufar.by/item/<id>` → 404/gone.
  Отсутствие в свежей выдаче ≠ снято (могло уехать за пределы первых N страниц).
- `stale=true, last_check=<now>` — если прямую проверку сделать не удалось / не делали.
- `defect=true` — defect-маркеры в title/body (negation-aware: «без дефектов»/«исправен» — не дефект).
- `no_price=true` — «договорная»/без цены → исключать из ранжирования (нельзя суммировать).
- Из истории **не удалять** — только пометки.

## 9. Нейминг файлов

```
[YYYY-MM-DD HHMM —] тема — тип.{md,json,csv,html}
```

- Дата + время вместе, только для date-bound файлов (отчёты, выгрузки, логи).
- **Время `HHMM` (24ч, без двоеточия) обязательно при наличии даты** — несколько прогонов
  в день не должны перетираться.
- Служебные/шаблонные (инструкции, чеклисты, этот файл) — без даты и времени.
- Разделитель — `—` (em dash с пробелами).
- Базовое имя совпадает во всех форматах, меняется только расширение.
- Подробно — `добавление-файлов — инструкция.md`.

## 12. Публикация на Caddy

`scripts/sync-to-caddy.sh` (запуск из корня проекта):
1. `md_to_html.py --all` — генерирует `*.html` из всех `reports/*.md`.
2. Через `ssh <remote-alias>` (LXC <caddy-ip>, ключ id_ed25519) заливает **только `*.html`**
   в `/var/www/<reports-dir>`, остальное подтирает.
3. `chown caddy:caddy`, права 644/755.

Сайт: `http://<reports-domain>` (IP-whitelist <internal-subnet>). На сервере только
HTML — `.md`/`.json`/`.csv` туда не попадают. Подробно — memory `project-caddy-kufar-reports-site`,
`project-caddy-lxc-ssh`.

## 14. Скрипты конвейера и SSOT — политика владения

Python-скрипты исторически хардкодили параметры (`kufar_filter.py` — DDR4/чипсеты/регион,
`kit_analyzer_run.py` / `reprocess_kits.py` — MB_BRANDS/INTEL_CHIPSETS/AMD_CHIPSETS/CPU_PERF/
baseline, `kufar_details.py` — defect-маркеры, `kufar_parse.py` — URL). С SSOT-ребрендингом
источником параметров стал `апгрейд-пк — пожелания.md`, и **синхронизацию скриптов с ним
делают сами субагенты**, а не вручную:

- **`kufar-listings-fetcher`** владеет `kufar_parse.py`, `kufar_details.py`,
  `transform_for_delivery.py`, `enrich_seen.py` — правит их под файл пожеланий (регион,
  URL категорий, defect-маркеры, no-price, свежесть).
- **`kit-analyzer`** владеет `kufar_filter.py`, `kit_analyzer_run.py`, `reprocess_kits.py`,
  `_build_report.py` — правит под файл пожеланий (DDR4/PCIe/чипсеты/бренды/CPU_PERF/
  baseline/тиры/cap'ы/веса/VRM).
- Принцип: если скрипт не отражает параметр из файла пожеланий или хардкодит устаревшее —
  субагент редактирует скрипт сам (предпочтительно — чтобы скрипт читал параметр из файла).
  Не удалять/не переименовывать; крупные правки — согласовывать с главным агентом одной строкой.
- `kufar_parse.py` обязан сохранять query в URL (фикс `project-kufar-parse-url-fix`) — не ломать при правке.

> ⚠️ Пока скрипты не переведены на чтение из файла пожеланий, правка файла меняет поведение
> agent-driven пути (через `kit-analyzer` субагента), но **не** script-driven пути
> (`/kufar-upgrade` → скрипты). Субагент закрывает этот разрыв при первом же прогоне, где
> замечает рассинхрон.

### Прочие пробелы
- **`/kufar-upgrade` SKILL.md** хардкодит URL с преднастроенными фильтрами (`ccch`/`ccrt`)
  и часть параметров — дубли с файлом пожеланий. SKILL направляет скрипты; при правке
  скриптов на чтение из файла SKILL тоже упрощается.
- **`reports/` содержит старые имена без времени** (`2026-07-04 — …` без HHMM) — до ввода
  обязательного времени. Не переименовывать без одобрения; новые отчёты — уже с HHMM.
- **`scripts/` clutter** — много `_*.py` / `_diag*.py` разовых скриптов. В конвейер входят
  только `kufar_parse.py`, `kufar_details.py`, `kufar_filter.py`, `_build_report.py`
  (+ `kit_analyzer_run.py`, `enrich_seen.py`, `md_to_html.py`, `sync-to-caddy.sh`).
  Остальные — исторические диагностические; не удалять без одобрения.
